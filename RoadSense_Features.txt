────────────────────────────────────────────────────────────────
  WHAT THE APPLICATION DOES
────────────────────────────────────────────────────────────────

RoadSense is a real-time road safety intelligence platform
designed to detect wrong-way drivers, predict collisions, and
guide drivers to safety. It combines a 3D driving simulation
with a live map, backend AI analysis, and multi-sensor data
fusion to replicate what an advanced driver assistance system
(ADAS) would do on real roads.

The app simulates an ego vehicle (your car) traveling through
Bengaluru, India, on a real road network. It monitors traffic
around you, detects anomalies, and alerts you instantly when
danger is detected.

────────────────────────────────────────────────────────────────
  WRONG-WAY DETECTION SYSTEM
────────────────────────────────────────────────────────────────

Two independent detection layers work together:

Layer 1 — Local Lane Analysis (frontend):
  - Counts same-direction vs opposite-direction vehicles
    in the ego's current lane within a proximity window
  - If more opposite-direction cars than same-direction → wrong lane
  - 1-second debounce prevents flickering false positives

Layer 2 — Traffic Flow Consensus (backend AI):
  - Analyzes bearings of all nearby vehicles (within 90 m)
  - Computes circular mean of peer headings (majority direction)
  - Triggers if ego bearing deviates > 140° from majority flow
  - Requires minimum 3 nearby vehicles at 10+ km/h
  - Confidence = (delta/180)*70 + majority_support*30
  - GPS noise guard: suppresses if GPS accuracy > 35 m AND
    confidence < 55%
  - Combined signal: wrongWayCombined = API.active OR localLane


────────────────────────────────────────────────────────────────
  BACKEND AI PIPELINE (/frontend-sim endpoint)
────────────────────────────────────────────────────────────────

Every 900 ms the frontend POSTs ego state + all vehicle positions.
The backend runs the full safety analysis pipeline:

1. Proximity filter — finds vehicles within 90 m radius
2. Traffic flow analysis — wrong-way detection via consensus
3. GPS noise guard — softens result if GPS accuracy is poor
4. Road passage recording — writes to SQLite (road_usage.db)
   with directional octant encoding (E0–E7)
5. Road intelligence query — historical dominant flow direction
   for the current road segment + time-of-day bucket
6. Flow opposition check — warns if user direction opposes
   historical majority flow
7. Collision bundle dispatch — if wrong-way active, computes
   Time-to-Collision (TTC), closing speed, and risk scores
   for nearby vehicles + synthetic pedestrians
8. Correction route suggestion — alternate path when wrong-way

Returns: wrong_way, collision_bundle, road_intelligence,
         road_warning, correction_route, time_slot


────────────────────────────────────────────────────────────────
  WRONG-WAY DETECTOR SIMULATOR (Backend Scenarios)
────────────────────────────────────────────────────────────────

3 predefined attack scenarios run on the backend road network
(Bengaluru road graph loaded from OpenStreetMap):

Scenario 1 — "The Near-Miss"
  Normal traffic + one high-speed wrong-way intruder
  Tests predictive alerting and TTC calculation

Scenario 2 — "The Diversion"
  Diversion traffic using wrong direction
  Tests crowd consensus suppression (should NOT alert)

Scenario 3 — "Urban Grid"
  Wrong-way vehicles on city streets (not highways)
  Tests detection on bidirectional road networks

Detection engine uses:
  - Heading mismatch threshold: 120°
  - Persistence window: 2.5 seconds before alert
  - Suspicion threshold: 80° at low speed (< 25 km/h)
  - Crowd consensus: 70% agreement over 60-second window
  - Alert radius: 500 m for at-risk vehicle scanning
  - Confidence scoring with attack classification


────────────────────────────────────────────────────────────────
  LIVE DATA CONNECTIONS
────────────────────────────────────────────────────────────────

REST API (http://127.0.0.1:8010):
  GET  /health          — backend status + road graph loaded
  GET  /scenarios       — list of 3 detector scenarios
  GET  /api/traffic     — live vehicle positions (polled every 500 ms)
  GET  /status          — simulator stats + bridge metrics
  POST /frontend-sim    — submit ego + vehicles, get safety analysis
  POST /start           — start a detector scenario
  POST /stop            — stop running scenario
  POST /consensus       — toggle crowd consensus mode
  GET  /road-intelligence/{id} — historical segment data
  POST /offline/predict — GPS-dead-reckoning prediction

WebSocket (ws://127.0.0.1:8010/ws):
  Real-time stream of:
  - tick: vehicle positions, new alerts, active alerts
  - safety_update: wrong-way + collision data
  - sim_ended: scenario finished

Traffic API response fields:
  vehicles[]: id, lane, t (0–1 route fraction), dir (1/-1),
              speed (km/h), position (meters)
  traffic: density ("low"/"medium"/"high"), alerts[]

────────────────────────────────────────────────────────────────
  ADDITIONAL FEATURES
────────────────────────────────────────────────────────────────

Auto Drive Mode:
  - Toggle button in top-right corner
  - Adaptive cruise control: follows car ahead (< 70 m gap)
  - Speed limit: 85 km/h
  - Smooth speed matching with lead vehicle

Route Planning:
  - Geocode Bengaluru landmarks by name
  - OSRM routing engine for real driving paths
  - Turn-by-turn steps extracted from route
  - Offline route caching (IndexedDB via offlineMode service)
  - Last sync time displayed

Offline Mode:
  - Detects navigator.onLine status
  - Caches route polyline for offline use
  - GPS dead-reckoning prediction via /offline/predict
  - Offline badge shown when disconnected

Map Layers:
  - Dark CARTO basemap
  - Route polyline (blue glow)
  - Route-ahead highlight (brighter blue, dynamic)
  - Ego vehicle dot (blue circle with white border)
  - Danger zone pulse (red halo when wrong-way/collision)
  - Detector vehicle dots (blue=normal, amber=suspicion, red=alert)
  - Alert markers for active wrong-way detections

Error Handling:
  - Traffic API offline → flushes vehicles, shows "OFFLINE" density
  - Backend unreachable → "offline" health state, graceful degradation
  - WebSocket disconnect → auto-reconnect after 2 seconds
  - Route geocode failure → user-friendly error message
  - All API calls wrapped in try/catch with UI feedback

Performance:
  - Three.js RAF loop decoupled from React render cycle
  - Map updates throttled to max 1 per 350 ms
  - Dead reckoning interpolates vehicle positions between API polls
  - Road geometry uses ribbon buffers (not recreated per frame)
  - NPC car pool reuse (no mesh recreation on respawn)
